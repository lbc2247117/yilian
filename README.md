# 驾培预约平台后端

驾培行业 O2O 平台，覆盖学员约课、教练接单、运营后台全流程，日均处理千级订单。基于 Kratos 构建的微服务架构，具备完善的服务治理与可观测性体系。

## 技术栈

Go · Kratos · gRPC · etcd · Jaeger · Redis · MySQL · Kafka · EFK · Prometheus · Docker · Kubernetes

## 架构设计

### 三层微服务拆分

```
┌─────────────────────────────────────────┐
│              BFF 层（3 个服务）            │
│  学员端 · 教练端 · 后台管理系统             │
├─────────────────────────────────────────┤
│          基础服务层（5 个服务）              │
│  用户 · 订单 · 系统配置 · 消息 · 第三方接口  │
├─────────────────────────────────────────┤
│           任务调度层                       │
│     定时任务 · 消息队列 Worker              │
└─────────────────────────────────────────┘
```

- **BFF 层**：面向不同客户端（学员小程序、教练小程序、PC 管理后台）提供定制化接口，完成数据聚合与格式适配
- **基础服务层**：封装核心领域逻辑，各服务独立数据库，通过 gRPC 对外暴露能力
- **任务调度层**：处理延时消息、定时结算、数据统计等离线任务

### 服务通信

- 服务间：**gRPC** 同步调用，Protocol Buffers 定义接口
- 对外：统一暴露 **HTTP** 端口，BFF 层完成协议转换
- 异步：**Kafka** 解耦订单状态变更、消息推送等事件驱动场景

### 服务治理

- **服务发现**：etcd 集群，支持健康检查与自动摘除
- **负载均衡**：客户端侧负载均衡，支持轮询、加权随机
- **限流**：基于令牌桶算法，BFF 层按客户端 + 接口粒度限流，基础服务层按调用方限流
- **熔断**：基于滑动窗口统计错误率，超过阈值自动熔断，半开探测恢复
- **重试**：指数退避 + 最大重试次数，仅对幂等接口开启

## 可观测性

### 链路追踪

全链路集成 **Jaeger**，gRPC 调用自动传播 Trace Context。每个服务启动时注册 Tracer，关键业务节点手动埋点（下单、支付、结算），单个请求可串联 BFF → 基础服务 → 数据库/缓存的所有调用。

### 指标监控

每个服务暴露两个标准端口：

- `/debug/pprof`：CPU 火焰图、内存分配、goroutine 分析
- `/metrics`：Prometheus 格式，覆盖 QPS、延迟分位数（P50/P90/P99）、错误率、gRPC 状态码分布、熔断器状态、限流拒绝数

Prometheus 定时拉取 → Grafana 仪表盘可视化 → AlertManager 异常告警。

### 日志体系

- 日志统一 **JSON 格式**输出到标准输出，字段包含 traceID、spanID、服务名、日志级别
- **Fluentd** 采集容器日志 → **Elasticsearch** 存储 → **Kibana** 面板检索
- 通过 traceID 在 Kibana 中一键串联同一请求的所有日志

## 数据库与缓存

- **MySQL**：订单、用户、车辆等核心业务数据，读写分离
- **Redis**：热点数据缓存（教练排期、城市车型配置）、分布式锁（结算任务互斥）、会话存储
- **ETag**：基于资源版本号的 HTTP 缓存协商，减少客户端重复拉取和网络传输
